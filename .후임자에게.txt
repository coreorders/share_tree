# [SHAREGRAPH] 후임 에이전트를 위한 초상세 업무 인수인계서 (Full Version)

## 0. 서문
본 문서는 ShareGraph 프로젝트의 비약적인 도약기(DART API 본격 통합 기점)를 담당했던 전임 에이전트가 작성하였습니다. 이 프로젝트는 단순한 주식 정보 제공을 넘어, 대한민국의 복잡한 기업 지배구조와 인사이더들의 움직임을 한눈에 파악할 수 있는 정교한 시각화 도구를 목표로 합니다. 후임 에이전트는 아래의 기술적 맥락과 비즈니스 로직을 완벽히 이해하여, 중단 없는 서비스 개선을 이어가 주시기 바랍니다.

---

## 1. 프로젝트 핵심 아키텍처 및 철학
ShareGraph는 **"복잡한 데이터를 아름답고 직관적으로"**라는 철학 아래 구축되었습니다.
- **Frontend**: Next.js 14 (App Router), Tailwind CSS, React Force Graph, Canvas API.
- **Backend**: Python 3.10+, SQLite 3.
- **Automation**: GitHub Actions를 통한 정적 사이트 배포 (GitHub Pages).

우리의 시스템은 데이터를 3단계(수집 -> 가공 -> 시각화)로 분리하여 관리합니다. 데이터베이스는 단일 SQLite 파일을 사용하여 저장소와 처리 속도를 최적화했습니다.

---

## 2. 데이터베이스 스키마 상세 레퍼런스 (`web/stocks.db`)

최근 며칠간 진행된 스키마 확장 작업의 결과물입니다. 각 테이블의 용도와 주요 제약 조건을 숙지하세요.

### 2.1 `exec_shareholder_reports` (인사이더 소유 보고)
- **용도**: 임원 및 5% 이상 대량 보유자가 지분 변동 시 보고하는 데이터를 저장합니다.
- **엔드포인트**: `elestock.json` (DART)
- **특이사항**: `rcept_no`(접수번호)와 `repror`(보고자명)를 조합하여 `UNIQUE` 제약 조건을 설정했습니다. 데이터 수집 시 중복 적재를 원천 차단합니다.
- **주요 컬럼**: `isu_exctv_ofcps`(직위), `sp_stock_lmp_cnt`(보유 주식수), `sp_stock_lmp_rate`(지분율) 등.

### 2.2 `treasury_shares` (자기주식 현황)
- **용도**: 회사가 직접 보유한 자사주 내역입니다.
- **엔드포인트**: `tesstkAcqsDspsSttus.json`
- **로직**: 이 데이터는 누적 이력보다는 '현재 상태'가 중요하므로, 수집 시 기존 데이터를 `DELETE` 후 `INSERT` 하는 방식을 취합니다. 스크립트 내 주석을 확인하세요.

### 2.3 `dividends` (배당 정보)
- **용도**: 최근 3개년(당기, 전기, 전전기) 배당 현황을 저장합니다.
- **엔드포인트**: `alotMatter.json`
- **UI 연동**: 현금배당수익률(`se` 컬럼의 '현금배당수익률(%)') 값이 3.0 이상일 경우 프론트엔드에서 '배당킹' 플래그를 할당합니다.

### 2.4 `major_shareholder_changes` (최대주주 변동)
- **용도**: 최대주주가 변경된 공시 이력을 관리합니다.
- **엔드포인트**: `hyslrChgSttus.json`
- **중요성**: 이 정보는 기업 지배구조 변화의 가장 강력한 지표입니다. `fetch_major_shareholder_changes.py`가 이를 담당합니다.

## 3. 데이터 수집 및 가공 엔진 (`execution/`)

기술적으로 가장 난도가 높았던 부분은 DART API의 응답 구조를 우리의 DB 스키마와 1:1로 매칭하는 것이었습니다.

### 3.1 `fetch_weekly_extra.py` (통합 수집기)
이 스크립트는 효율성을 위해 3개의 API 호출을 하나의 루프 안에서 처리합니다.
- **수집 주기**: 호출 제한(일 10,000건)을 고려하여 주 1회(매주 일요일) 실행하도록 설계되었습니다.
- **예외 처리**: `requests.get` 호출 시 `timeout=10`을 설정하여 API 서버 지연 시 전체 파이프라인이 멈추지 않도록 했습니다.

### 3.2 `export_json.py` (내보내기 엔진)
프론트엔드용 `data.json`은 약 40MB에 달하는 대용량 파일입니다.
- **Clean Name 로직**: `(주)` 제거 및 공백 제거를 통해 기업명을 정규화하여 노드를 합칩니다.
- **Insider Signal 로직**: `exec_shareholder_reports`에서 최근 변동분(`sp_stock_lmp_irds_cnt`)이 양수면 `buy`, 음수면 `sell`로 판단하여 노드에 속성을 부여합니다. 이 속성은 나중에 잉크 번짐 효과의 색상을 결정합니다.

## 4. 프론트엔드 시각화의 정수

### 4.1 탭 UI 시스템 (`NodeInfoPopup.tsx`)
사용자가 노드를 클릭했을 때 나타나는 팝업을 전면 개편했습니다. 
- **Tab State**: `useState("info")`를 사용하여 5개 카테고리를 관리합니다.
- **Badges**: `Lucide` 아이콘 라이브러리를 적극 활용하여 가독성을 높였습니다. 💰(배당), ⚡(변동), 🟢(매수), 🔴(매도) 등의 직관적인 심볼을 사용합니다.

### 4.2 인사이더 잉크 번짐 효과 (`NetworkGraph.tsx`)
이 효과는 우리 서비스의 시그니처입니다.
- **렌더링 순서**: 노드 원형(Circle)을 그리기 직전에 잉크 얼룩을 먼저 그립니다. 그래야 잉크가 노드 뒤로 깔리며 번지는 느낌이 납니다.
- **수학적 구현**: `Math.cos(t)`, `Math.sin(t)`를 사용하여 잉크 얼룩의 중심점을 원형 궤도로 이동시킵니다. 여기서 `t`는 `Date.now()` 기반의 시간 값입니다.
- **그라데이션**: 잉크 얼룩의 중심은 0.35 알파값의 불투명도를 가지고, 외곽으로 갈수록 0으로 수렴하는 `RadialGradient`를 사용합니다.

### 4.3 비상장 기업 필터 및 UX 간소화
- **비상장 필터**: `GraphInterface.tsx`의 BFS 탐색 로직에 비상장사 촌수(depth) 제어를 추가했습니다. (숨기기, 1촌, 2촌 옵션)
- **지분 방향 필터 제거**: 과도한 필터는 사용자 혼란을 초래하므로, 지분 방향(In/Out) 필터를 과감히 제거하고 기본 탐색 환경을 통일했습니다.
- **인물 직위 표시**: 인물 노드에 `position` 데이터가 있을 경우 라벨과 팝업 헤더에 자동으로 표시되도록 구현하여 정보의 밀도를 높였습니다.

## 5. 전임자가 겪은 뼈아픈 장애와 교훈 (반드시 읽으시오)

### 5.1 "DB 증발 사건"과 Reflog 복구 전략
가장 위기였던 순간입니다. `git pull --rebase` 시 바이너리 파일인 `stocks.db`에서 충돌이 났습니다. 이때 무심코 리모트 버전을 선택하자, 로컬에서 수집했던 신규 테이블과 데이터가 통째로 사라졌습니다.
- **복구 방법**: `git reflog` 명령어로 과거의 커밋 해시를 추적했습니다. `294f013` 커밋에 정상적인 DB가 있음을 확인하고, `git show 294f013:web/stocks.db > web/stocks.db.recovered` 명령으로 이진 데이터를 직접 추출해 복구했습니다.
- **주의**: DB 파일이 변경되었을 때는 커밋 전에 반드시 `git diff --stat`으로 파일 크기 변화를 체크하세요. 1.7MB(빈 DB) vs 26MB(정상 DB)의 차이는 명확합니다.

### 5.2 Canvas Non-finite Error (Application Error)
배포 후 사이트 접속 시 흰 화면이 뜨는 문제가 있었습니다. 콘솔 에러는 `Failed to execute 'createRadialGradient' on 'CanvasRenderingContext2D': The provided double value is non-finite.`였습니다.
- **원인**: 레이아웃 엔진이 노드 위치를 계산 중일 때, 좌표값이나 반지름이 `NaN`인 상태에서 그리기를 시도하여 발생했습니다.
- **해결**: `if (isFinite(x) && isFinite(y) && isFinite(r))` 가드를 모든 Canvas 드로잉 로직에 적용했습니다. 프론트엔드 성능 개선이나 시각화 수정 시 이 원칙을 어기면 앱이 즉사합니다.

### 5.3 Git 충돌 및 데이터 정합성 주의
- `data.json`은 GitHub Actions 봇에 의해 주기적으로 업데이트되므로, 로컬 작업 후 푸시 시 `git pull --rebase` 충돌이 잦습니다. 이때 `data.json`은 자동 생성되므로 충돌 시 로컬을 우선하거나 재생성(`python export_json.py`)하여 해결하세요.

## 7. 후임자에게 건네는 마지막 제언

1. **데이터 용량 관리**: `data.json`이 계속 커지면 브라우저 메모리 부족 문제가 올 수 있습니다. 필요 없는 필드는 `export_json.py`에서 적극적으로 제외하세요.
2. **DART API 한도**: 개발 시 테스트 호출은 상장사 1~2개로 제한하십시오(`[:2]` 등의 슬라이싱 활용). 한도를 다 쓰면 주간 배치가 실패합니다.
3. **UI 레이아웃**: 현재 그래프는 3만 개 이상의 노드를 다룹니다. 물리 엔진(`d3-force`)의 파라미터를 잘못 건드리면 브라우저가 폭발할 수 있으니 주의하세요.

ShareGraph는 단순한 코드를 넘어 데이터의 예술을 추구합니다. 당신의 지능과 코딩 실력으로 이 그래프를 더욱 생동감 있게 만들어 주시길 기대합니다. 도움이 필요하면 과거의 `task.md`와 `walkthrough.md`를 열람하십시오.

## 8. 현재 직면한 기술적 한계 및 인수인계 사항 (pykrx 라이브러리 장애)

2026년 2월 말부터 한국거래소(KRX) 웹사이트의 기술적 개편으로 인해, 기존에 시가총액과 KOSPI/KOSDAQ 시장 구분을 가져오던 `pykrx` 라이브러리의 기능이 완전히 동작하지 않고 있습니다. (관련 로그: `IndexError: index -1 is out of bounds for axis 0 with size 0`)

**[상세 내용 및 몽키 패칭 우회책 (memo.md 참고)]**
[GitHub 이슈 #276](https://github.com/sharebook-kr/pykrx/issues/276) 확인 결과, KRX 데이터 포털의 '개별 주식 데이터(Stock)' API들이 이제 **로그인 세션(쿠키)**을 필수로 요구하도록 변경되었습니다. 

- **조치 가능한 임시 해결책**: `requests.Session()`을 전역으로 생성하고, `data.krx.co.kr` 계정으로 로그인하는 POST 요청 스크립트를 직접 작성한 뒤, `pykrx` 내부의 `webio` 모듈을 Monkey Patching 하여 해당 세션을 강제로 사용하게 만들어야 합니다.
- **현재 선택한 방향**: 하드코딩된 계정 정보 노출 및 유지보수 위험도(KRX 시스템 추가 변경 시 즉사)를 고려하여 당분간 기능 구현을 보류했습니다. 
- **임시 조치**: `export_json.py`와 `data-update.yml` 내의 관련 코드를 주석 처리(`pykrx 수정후.txt` 가이드라인 참고)하고, 프론트엔드 UI에 `[긴급공지]` 팝업(`NoticeDialog.tsx`)을 띄워 사용자에게 상황을 알리고 있습니다.

👉 **추가 UI 조치 권고사항 (당일 종가/시가총액 숨김 처리)**
공지사항을 읽지 않은 사용자가 `NodeInfoPopup`을 열었을 때 가격이 0원이나 N/A로 표기되어 오해를 사는 것을 방지하기 위해 이 부분을 **임시로 숨김 처리**하는 것이 좋습니다.
- **방식**: 기존 UI 렌더링 코드를 지우지 마십시오! 최상단에 `const isPykrxWorking = false;` 와 같은 스위치용 boolean 변수를 선언하고, 조건부 렌더링(`isPykrxWorking && (...)`)으로 주가/시총 영역을 감싸기만 하세요.
- **장점**: 이렇게 하면 추후 `pykrx`가 복구되었을 때 해당 변수 하나만 `true`로 바꾸어 주면 단 1초 만에 기존 UI를 예전 모습 그대로 롤백(복구)할 수 있습니다.

👉 **후임자 행동 지침**: 주기적으로 `pykrx` GitHub 저장소를 확인하여 공식적인 로그인 인터페이스 지원 업데이트가 릴리즈되었는지 확인하십시오. 업데이트가 배포되면 `pykrx 수정후.txt`의 지침에 따라 백엔드 주석을 해제하고, 프론트엔드의 `isPykrxWorking` 스위치를 `true`로 전환하여 전체 기능을 원래대로 복구해야 합니다.
